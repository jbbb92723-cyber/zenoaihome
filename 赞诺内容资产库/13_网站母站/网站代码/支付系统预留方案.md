# 赞诺内容母站 — 支付系统预留方案

> 文档版本：v1.0 · 2026-04-20  
> 第一阶段：只建结构，不接真实支付  
> 第二阶段：接入 Stripe / 微信支付 / 支付宝

---

## 一、支付系统未来要解决什么

| 业务问题 | 支付系统如何解决 |
|---|---|
| 用户想购买付费资料 | 创建订单 → 调起支付 → 支付成功开权限 |
| 用户想开通付费会员 | 选择会员套餐 → 支付 → 更新 memberships 表 |
| 赞诺想查看收入 | 订单表记录每笔交易 |
| 支付失败怎么处理 | 订单状态标为 failed，不开权限 |
| 已开权限但申请退款 | 关闭 memberships 或下载权限 |
| 用户重复支付 | 幂等处理 + provider_order_id 唯一约束 |

---

## 二、支付和会员权限的关系

```
用户发起支付
    ↓
创建 orders 记录（status: pending）
    ↓
跳转支付平台（Stripe / 微信 / 支付宝）
    ↓
【前端不能判断支付成功！】
    ↓
支付平台推送 webhook 到 /api/payments/webhook
    ↓
服务端验签（HMAC）
    ↓
验签通过 → 更新 orders.status = paid
    ↓
根据 product_type 开通权限：
  - membership → 创建/更新 memberships 记录
  - resource   → 标记 resource_downloads 权限
    ↓
用户刷新页面 → 权限已开通
```

## 三、订单表如何工作

| 字段 | 说明 |
|---|---|
| id | 内部订单 ID（UUID）|
| user_id | 谁下的单 |
| product_type | 购买的是什么：membership / resource / service |
| product_id | 具体是哪个产品（member tier 或 resource UUID）|
| amount | 金额（单位：分，避免浮点问题）|
| currency | CNY / USD |
| status | pending / paid / failed / refunded / canceled |
| payment_provider | stripe / wechat / alipay / manual |
| provider_order_id | 支付平台订单 ID（用于对账和防重）|
| paid_at | 实际支付时间（webhook 确认后写入）|

## 五、支付回调为什么要验签

支付平台在支付成功后会向你的服务器发送 HTTP 请求（webhook）。  
**如果不验签：** 任何人都可以伪造一个请求告诉你「某用户支付成功了」，然后免费获得权限。  
**验签原理：** 支付平台用共享密钥对请求内容做 HMAC 签名，你用同一密钥验证签名是否一致。

```typescript
// Stripe 示例：
const event = stripe.webhooks.constructEvent(
  rawBody,                                    // 原始请求体
  request.headers.get('stripe-signature')!,  // 平台签名
  process.env.STRIPE_WEBHOOK_SECRET!,        // 你的 webhook 密钥
)
// 如果签名不匹配，会抛出异常，拒绝处理
```

## 六、为什么不能只靠前端判断支付成功

1. **前端代码可以被篡改。** 任何人都可以在浏览器控制台修改变量。
2. **网络请求可以被拦截和伪造。** 如果前端发送「支付成功」请求，可以轻易伪造。
3. **支付平台的 webhook 是真实的支付凭证。** 必须用服务端验签确认。
4. **幂等保证。** webhook 可能多次推送同一事件，服务端需要防止重复开权限。

---

## 七、第一阶段先不接真实支付的原因

1. **避免过早复杂化。** 支付涉及合规、密钥管理、退款流程，不是一两天能做好的。
2. **先跑通权限模型。** 在没有支付的情况下，先验证权限逻辑是否正确。
3. **减少安全风险。** 没有支付就没有金融风险，出问题只是功能缺失，不是资金损失。
4. **第一阶段没有真实流量。** 没有流量，就不需要真实支付。

## 八、第二阶段可接入的支付方式

| 支付方式 | 适用场景 | 接入难度 | 国内合规 |
|---|---|---|---|
| **微信支付** | 国内用户，移动端为主 | 中（需企业资质）| ✅ |
| **支付宝** | 国内用户，与微信并行 | 中（需企业资质）| ✅ |
| **Stripe** | 国际用户，卡支付 | 低（文档好）| ⚠️（境外收款需注意）|
| **手动转账审核** | 早期小规模，最简单 | 极低 | ✅ |

**推荐顺序：** 手动转账审核 → Stripe → 微信支付/支付宝

### 手动转账审核方案（最简单，第一阶段可用）

1. 用户填写购买申请表（姓名 + 联系方式 + 产品）
2. 赞诺提供收款二维码（微信/支付宝个人收款）
3. 用户截图付款凭证，提交到联系表单
4. 赞诺手动审核，在 Supabase 后台修改 orders.status = paid 和 profiles.role
5. 无需 webhook，无需 API

> 优点：零代码，零风险，适合早期验证  
> 缺点：无法自动化，需要人工处理

## 九、当前阶段怎么模拟付费资源

第一阶段用管理员直接修改数据库中的用户 role 来模拟：

1. 在 Supabase 管理面板 → Table Editor → profiles 表
2. 找到目标用户记录
3. 将 role 改为 `member` 或 `customer`
4. 用户下次访问时即可看到对应权限的内容

这样可以在不接支付的情况下，测试整个权限流程是否正确。
