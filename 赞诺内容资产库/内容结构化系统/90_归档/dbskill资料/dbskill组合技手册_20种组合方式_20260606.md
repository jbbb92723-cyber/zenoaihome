这份手册整理了 dbskill 的 20 种组合用法，把单个 skill 串成「目标澄清、商业诊断、执行诊断、内容生产、决策沉淀」等工作流。核心判断是：一个 skill 的输出经常可以作为另一个 skill 的输入，两步是入门，五步以上可以形成完整连技。

入门组合详解
1. 目标净化：/dbs-goal → /dbs-diagnosis
「我想做个人 IP」进诊断，和「六个月内让付费咨询月流水从零到三万」进诊断，是两件完全不同的事。/dbs-goal 做的事情是把模糊的目标变成你能检查的东西——它没有完成的时候你知道，它完成的时候你也知道。带着这样的目标进诊断，诊断才有锚点

2. 问题重构：/dbs-good-question → /dbs-diagnosis
「为什么我的内容没人买」这个问题没法回答，因为它没有指向任何可以观察的现象。/dbs-good-question 把它变成「最近三个月私信咨询从每周二十条降到三条，但发布频率没变」——同一个困惑，换个问法，诊断才能对准真正的东西

3. 目标查执行：/dbs-goal → /dbs-action
很多执行不动，根本原因是目标本身就是虚的。模糊的目标产生不了真实的驱动力，于是「不做」变成了理性选择，不是懒。先用 /dbs-goal 把目标变成真实可检查的，再用 /dbs-action 看执行障碍，顺序不要反

4. 外+内诊断：/dbs-diagnosis → /dbs-action
商业模式是外部问题，心理逃避是内部问题，两个可以同时存在。/dbs-diagnosis 检查商业模式有没有结构性缺陷，/dbs-action 用阿德勒框架诊断「知道该做什么但就是不做」的真正原因。两个都跑，再对比结论，通常能搞清楚症结在哪

5. 结论压测：/dbs-diagnosis → /dbs-chatroom
单次诊断是单一视角，有局限。诊断出一个结论之后，把这个结论丢进聊天室，让不同立场的角色来打压力测试。能经过多角色质疑的结论才值得信，经不住的说明还有漏洞

6. 内容做开头：/dbs-content → /dbs-hook
内容方向和结构没定之前，开头没法做好——开头是内容的预告，预告一个没想清楚的内容，五秒之内就露馅了。先用 /dbs-content 诊断这期内容该怎么做，结构和切角确定了，再用 /dbs-hook 做开头

7. 开头做标题：/dbs-hook → /dbs-xhs-title
开头完成后顺手做小红书标题，不需要重新想。开头的第一句话本来就可以直接变成标题，两步之间有天然的内容复用，顺序跑就行

8. 诊断建档：/dbs-diagnosis → /dbs-decision
/dbs-diagnosis 的结论漂在聊天里，对话关掉就消失了。/dbs-decision 把它变成一个持续维护的本地文件，事实、规律、阶段状态、还没想通的问题，分层存放，随时能接着用。这个组合的意义是时间维度的——把单次诊断变成长期追踪

进阶组合详解
9. ★ 三步前处理：/dbs-goal → /dbs-good-question → /dbs-diagnosis
这是我认为最容易被忽略、ROI 最高的组合

大部分人进诊断之前不做任何准备，带着一个模糊问题直接进去，跑完一圈感觉有点收获但又说不清收获在哪。原因很简单：输入是虚的，诊断只能在空气上打转

这三步依次解决三个不同的输入问题：/dbs-goal 让你搞清楚自己到底想要什么，/dbs-good-question 让你把困惑变成一个可以观察的现象，带着这两样东西进 /dbs-diagnosis，诊断才有东西可以对准

举个例子：你觉得自己的变现效率很低。先用 /dbs-goal 审计——「提高变现效率」是什么意思，具体到什么数字算达到了？然后用 /dbs-good-question 把问题变成「过去六个月平均每个私信用户的成交率是 3%，同类博主的数据大概是 12%，差距主要出现在哪个环节」。带着这个问题进诊断，结果是完全不同的

10. 目标→模式→执行：/dbs-goal → /dbs-diagnosis → /dbs-action
三步覆盖三个层次：你要去哪里、商业模式能不能带你去、你有没有在走。任何一个层次出问题，结果都是「做不动」，但原因完全不同。这个组合把三个层次都检查一遍，结论出来之后你会清楚症结在哪一层

11. ★ 执行困境三角：/dbs-diagnosis + /dbs-action + /dbs-slowisfast（并联）
这是 20 种组合里唯一一个并联结构——三个 skill 不是按顺序跑，是同时从三个方向攻同一个问题

「我不知道为什么做不动」这个问题有三个可能的根因：商业模式有结构缺陷（/dbs-diagnosis）、心理在有目的地逃避（/dbs-action）、选了一条看起来快但长期反噬的路（/dbs-slowisfast）。三个可能同时存在，也可能只有一个

用法是：三个 skill 分别跑，把三份结论放在一起看。重叠的部分就是根因，只出现在一个 skill 里的可能是表象

12. 内容小三角：/dbs-benchmark → /dbs-content → /dbs-hook
内容诊断需要参照系，凭空诊断方向容易跑偏。先用 /dbs-benchmark 找到值得模仿的对标，带着「他是怎么做内容的」这个问题进 /dbs-content 诊断，再用 /dbs-hook 做开头。三步之间有明确的依赖关系，顺序不能换

13. 概念转内容：/dbs-deconstruct → /dbs-content → /dbs-hook
想做关于某个概念的内容，最大的坑是这个概念你自己还没说清楚。先用 /dbs-deconstruct 把核心词拆明白——它在你的语境里到底是什么意思、有几种用法、大多数人的误解在哪——再进内容诊断，方向才不会虚

高级组合详解
14. 商业诊断完整链：/dbs-goal → /dbs-good-question → /dbs-diagnosis → /dbs-chatroom
前两步做输入质量，第三步做核心诊断，第四步做结论验证。跑完这四步，你拿到的不是一个 AI 生成的答案，而是一个经过目标锚定、问题重构、单视角诊断、多视角压测之后的结论。适合用在重要的商业判断上

15. 执行困境完整链：/dbs-goal → /dbs-diagnosis → /dbs-action → /dbs-slowisfast
四步覆盖四个维度：目标是不是真实的、商业模式有没有问题、心理在不在逃避、方法路径有没有选对。这四个维度各自独立，但最终的症状都是「做不动」。适合在一段时间内反复跑不起来、原因又不清楚的时候用

16. ★ 内容生产完整链：/dbs-benchmark → /dbs-content → /dbs-hook → /dbs-ai-check
从「找到参照系」到「内容可以发出去」的完整过程

/dbs-benchmark 给你一个参照系，你知道同类内容里谁做得好、他的内容有什么结构。/dbs-content 在这个参照系里诊断你的内容该怎么做，形式对不对，切角对不对。/dbs-hook 在方向确定之后做开头，开头必须在五秒内独立建立吸引力。/dbs-ai-check 是最后一道质检，拿到具体的检测报告，再决定要不要改

这四步每一步都依赖上一步的输出。跳过任何一步不是不行，但跳过之后做出来的内容，质量差距是可以感觉到的

17. 概念到内容全程：/dbs-deconstruct → /dbs-learning → /dbs-content → /dbs-hook
适合想把自己对某个领域的理解做成系列内容的人。先把核心词拆清楚，再用 /dbs-learning 把这个领域做成系列学习文章（这个过程会让你发现之前没想到的角度），再用 /dbs-content 诊断哪篇适合做内容、怎么做，最后做开头。这条链的特点是，每一步都在让你对这个课题的理解更扎实

连技详解
18. ★ 内容发布全链：/dbs-benchmark → /dbs-content → /dbs-hook → /dbs-ai-check → /dbs-xhs-title
五步覆盖从「找对标」到「内容可以发」的全程

/dbs-benchmark 找到值得模仿的对标，/dbs-content 在这个参照系里诊断内容该怎么做，/dbs-hook 在方向确定后做开头，/dbs-ai-check 做最后一道质检，/dbs-xhs-title 生成小红书标题

这条链和「内容生产完整链」（第 16 条）的区别在于最后一步：完整链到 AI 检测就结束，发布全链多了标题这一步，跑完之后内容是真的可以发出去的状态。适合想一次性把一条内容从诊断做到发布就绪的场景

19. ★ 商业决策全链：/dbs-goal → /dbs-good-question → /dbs-diagnosis → /dbs-chatroom → /dbs-decision
这条链的终点是 /dbs-decision，这是它和其他诊断类组合最大的区别

前四步是充分的准备：目标清晰、问题具体、诊断扎实、结论经过多角色压测。但这四步跑完，结论还是漂在聊天里的。/dbs-decision 把整个过程的输出变成一个本地的、分层的、可以持续维护的决策档案——事实层、规律层、状态快照、待解问题，各放各的位置

适合用在需要长期跟踪的商业判断上，比如要不要做一个新产品线、要不要和某个人合作

20. 决策沉淀全链：/dbs-goal → /dbs-diagnosis → /dbs-decision → /dbs-save × n → /dbs-report
这条链的特点是时间跨度最长，不是单次对话能跑完的

起点是目标清晰和诊断，建立 /dbs-decision 档案之后，每次新的诊断或判断就用 /dbs-save 存档。积累几次之后，/dbs-report 把所有存档合并成一份带时间线、可以分享的报告。这份报告可以拿去给合伙人看，做季度复盘，或者交给外部顾问作为背景材料
