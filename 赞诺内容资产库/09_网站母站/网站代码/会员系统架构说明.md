# 赞诺内容会员系统 — 架构说明

> 文档版本：v1.0 · 2026-04-20  
> 本文档是赞诺内容母站会员系统的技术架构说明，供开发参考和阶段推进使用。

---

## 一、为什么要做登录和会员

这不是普通的博客留言系统升级，而是赞诺内容资产系统的核心基础设施建设。

| 业务问题 | 登录/会员如何解决 |
|---|---|
| 资料被随意爬取 | 登录门槛隔离非真实用户 |
| 无法识别谁是真实读者 | 用户系统沉淀真实受众 |
| 服务转化全靠微信 | 会员体系支持在线收款和权限管理 |
| 资料没有差异化价值 | 付费/会员资料与免费资料隔离 |
| 无法管理历史订单 | 订单系统支持服务追踪 |
| 评论无法管理 | 审核机制过滤垃圾信息 |
| 没有用户行为数据 | 可积累资料下载、阅读数据 |

**核心判断：**  
用户系统不是为了做成 SaaS，而是为了让内容资产产生可积累、可变现的持续价值。  
先建底座，不要一步做复杂。

---

## 二、这套系统的功能边界

### 在系统范围内：
- 用户登录 / 注册（OAuth + 邮箱）
- 基础会员状态（免费/付费）
- 资料访问权限控制
- 评论发布与审核
- 视频索引（外链托管）
- 订单记录（支付后台预留）
- 用户中心（个人信息 + 已领资料 + 订单）

### 不在范围内（此阶段）：
- 复杂后台 CMS
- 自动化课程系统
- 真实支付接入（第一阶段预留接口即可）
- 微信生态深度集成（复杂，单独规划）
- 多级代理/分销机制

## 五、为什么不一开始就做复杂后台

1. **复杂后台是产品重心，不是内容重心。** 赞诺的核心价值是内容和信任，而不是功能堆砌。
2. **早期没有真实流量，做复杂后台是浪费。** 先把登录和权限底座跑通，再根据真实需求迭代。
3. **维护成本高。** 复杂系统意味着更多 bug，早期不值得。
4. **第一版后台可以是 Supabase 自带的管理界面。** 不需要重新开发。

### 需要安装的依赖（第一阶段）

```bash
# Prisma ORM
npm install prisma @prisma/client
npx prisma init

# 或者使用 Supabase 客户端（不用 ORM）
npm install @supabase/supabase-js
```

**当前状态：**  
第一阶段代码使用占位 mock 函数，不强依赖真实数据库连接。  
配置好 `DATABASE_URL` 后再初始化 Prisma 即可。

## 七、OAuth / IDC Flare 接入逻辑

### 流程图

```
用户点击「IDC Flare 登录」
    ↓
浏览器跳转到 IDC Flare 授权页面
    ↓
用户在 IDC Flare 确认授权
    ↓
IDC Flare 回调：/api/auth/callback/idcflare
    ↓
Auth.js 用 code 换取 access_token
    ↓
Auth.js 用 token 请求 userinfo endpoint
    ↓
Auth.js 创建 session（JWT 或数据库 session）
    ↓
用户登录成功，跳转 /account
```

### 关键配置项（待填写）

```env
IDCFLARE_CLIENT_ID=           # 在 IDC Flare 应用后台获取
IDCFLARE_CLIENT_SECRET=       # 在 IDC Flare 应用后台获取
IDCFLARE_ISSUER=              # 【待补充：IDC Flare OAuth issuer URL】
IDCFLARE_AUTHORIZATION_URL=   # 【待补充：授权页面 URL】
IDCFLARE_TOKEN_URL=           # 【待补充：token 换取 URL】
IDCFLARE_USERINFO_URL=        # 【待补充：用户信息 URL】
```

> ⚠️ **IDC Flare 提示：**  
> IDC Flare 是否提供标准 OIDC（OpenID Connect）接入，尚需确认官方文档。  
> 如果只提供 OAuth2，需要手动指定 `authorization_url / token_url / userinfo_url / scopes`。  
> 详见 `IDC-Flare-OAuth接入说明.md`

## 十一、资料下载权限规划

| 资料类型 | access_level | 操作 |
|---|---|---|
| 免费公开资料 | public | 直接查看/下载链接 |
| 需登录资料 | login | 登录后领取 |
| 会员资料 | member | 开通会员后领取 |
| 付费资料 | paid | 购买后下载 |
| 管理员专属 | admin | 不对外显示 |

**第一阶段：** 前端展示权限标签和按钮逻辑，下载本身仍通过公众号引导。  
**第二阶段：** 接入真实文件存储，按权限生成临时下载链接。

## 十二、视频系统规划

视频不存储在网站服务器，只保存外部链接和封面图。  
支持平台：B站 / YouTube / 视频号 / 腾讯云 COS / Cloudflare R2

每条视频记录包含：
- title / slug / description
- platform / url / embedUrl
- coverImage / accessLevel
- relatedArticleSlug（关联文章）

**第一版：** 只做视频索引页，展示封面 + 跳转链接。  
**第二版：** 支持 iframe 嵌入和会员权限。

## 十四、安全注意事项

| 风险 | 应对方案 |
|---|---|
| 密钥泄露 | 所有密钥放 .env.local，不提交 git |
| CSRF 攻击 | Auth.js 内置 CSRF 防护 |
| 支付回调伪造 | 验签（hmac），不依赖前端判断 |
| SQL 注入 | 使用 Prisma ORM 参数化查询 |
| XSS 攻击 | 评论内容过滤 HTML，不直接 dangerouslySetInnerHTML |
| 暴力破解 | Auth.js 内置 rate limiting（配置后生效）|
| 下载链接泄露 | 生成临时签名 URL（第二阶段）|
| CORS 跨域 | API 路由默认限制同源，webhook 单独处理 |

## 十五、部署到 Vercel 后如何配置环境变量

1. 打开 Vercel Dashboard → 项目 → Settings → Environment Variables
2. 逐个添加以下变量：

```
AUTH_SECRET           = （随机生成，至少 32 位，openssl rand -base64 32）
AUTH_URL              = https://你的域名.com
IDCFLARE_CLIENT_ID    = （从 IDC Flare 后台获取）
IDCFLARE_CLIENT_SECRET = （从 IDC Flare 后台获取）
IDCFLARE_AUTHORIZATION_URL = 【待补充】
IDCFLARE_TOKEN_URL    = 【待补充】
IDCFLARE_USERINFO_URL = 【待补充】
DATABASE_URL          = （Supabase 连接字符串，在 Supabase 项目 Settings → Database 中获取）
NEXT_PUBLIC_SITE_URL  = https://你的域名.com
```

3. 添加完毕后，重新部署（Redeploy）即可生效。

> **重要提示：**  
> `AUTH_SECRET` 是 Auth.js 用于签名 session token 的密钥。  
> 每个环境（本地/生产）建议使用不同的值。  
> 可以用命令生成：`openssl rand -base64 32`
