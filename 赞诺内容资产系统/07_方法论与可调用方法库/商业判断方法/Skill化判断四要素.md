## 原始输入

输入 + 规则 + 输出 + 高频 = 值得 skill 化。

## 一句话提炼

一个任务是否值得 skill 化，不看它复杂不复杂，而看它是否有稳定输入、明确规则、可验收输出，并且高频发生。

## 适用场景

用于判断一个流程、判断框架、内容处理动作、客户交付动作，是否值得封装成：

- skill；
- 智能体；
- SOP；
- prompt；
- 方法卡；
- 自动化流程。

## 四个必要条件

### 1. 输入稳定

问：

- 每次开始时，输入材料是否相似？
- 输入能否被描述清楚？
- 是否可以列出必填信息？
- 缺信息时，是否知道要追问什么？

如果输入每次都完全不同，先不要 skill 化，先做分类。

### 2. 规则明确

问：

- 判断标准是否清楚？
- 步骤是否可以写出来？
- 有哪些禁区和边界？
- 什么情况要人工判断？
- 什么情况直接拒绝或归档？

没有规则的 skill，只会把混乱自动化。

### 3. 输出可验收

问：

- 最终产物是什么？
- 好坏标准是什么？
- 输出格式是否固定？
- 人能不能快速检查它是否合格？
- 输出是否能进入下一步流程？

输出不可验收，就不能放心复用。

### 4. 高频发生

问：

- 这个任务是否经常出现？
- 每次手动处理是否消耗注意力？
- 未来是否还会反复遇到？
- 封装后是否能明显降低下一次成本？

低频任务可以写笔记，不一定要 skill 化。

## 判断公式

> 输入稳定 + 规则明确 + 输出可验收 + 高频发生 = 值得 skill 化。

缺一项时：

- 缺输入：先做输入模板；
- 缺规则：先做方法卡；
- 缺输出：先定义验收标准；
- 缺高频：先保存为普通笔记或案例。

## 输出判断

### A 档：立即 skill 化

四项都满足。

动作：写成 skill / prompt / agent，并加入可调用方法库。

### B 档：先方法卡化

有稳定思路，但规则或输出还不够清楚。

动作：先写方法卡，多跑几次后再 skill 化。

### C 档：先保留为素材

低频、输入混乱、规则不清。

动作：先进入旧观点 / 案例 / 灵感库，不急着封装。

## Zeno OS 应用

适合 skill 化的任务：

- 外部观点收进知识库；
- 爆款链接采集后的对标筛选；
- 旧观点检索；
- 素材膨胀；
- 内容资产生产单；
- ZenoAIHome 报价风险初筛；
- OPC 行业生态位判断；
- AI 工具/课程是否值得学的判断。

不适合直接 skill 化的任务：

- 还没想清楚的战略判断；
- 高度依赖 Zeno 个人审美的最终文案；
- 低频一次性项目；
- 输入材料极其混乱、分类尚未建立的任务。

## 可调用提示

> 请用「Skill化判断四要素」判断这个任务是否值得 skill 化：
> 
> - 输入是否稳定；
> - 规则是否明确；
> - 输出是否可验收；
> - 是否高频发生；
> - 缺哪一项；
> - 应该立即 skill 化、先方法卡化，还是先保留为素材。

## 不适合做什么

- 不适合把所有想法都封装成 skill。
- 不适合用 skill 替代人的最终判断。
- 不适合把模糊流程自动化。
- 不适合低频任务过度工程化。

## 一句话

Skill 化的前提不是“看起来高级”，而是这个任务已经有稳定输入、明确规则、可验收输出，并且反复发生。
