## 适用场景

用于判断一个 AI 应用想法，应该当成自用提效工具、兴趣项目、服务交付组件，还是长期商业产品。

尤其适合一人公司、传统行业老板、AI 培训/咨询场景，避免把“能做出来”误判成“能长期赚钱”。

## 输入

- 一个 AI 应用 / 自动化 / 小工具想法；
- 它解决的问题；
- 目标使用者；
- 是否有人愿意付费；
- 稳定性和交付责任要求；
- 当前开发与维护能力。

## 双轨判断

### A 轨：自用提效 / 兴趣工具

适合条件：

- 主要解决自己的真实问题；
- 不需要对外承诺稳定性；
- 出 bug 的损失可控；
- 可以快速迭代；
- 价值来自节省自己时间、整理自己流程、增强自己能力。

判断标准：

- 我自己会不会每天/每周用？
- 它能不能减少一个真实摩擦？
- 失败了会不会造成严重损失？
- 它能不能沉淀成案例、SOP 或方法卡？

适合动作：

> 快速做，快速用，快速改。不要过早包装成产品。

### B 轨：长期盈利产品

适合条件：

- 有明确付费人群；
- 问题长期存在；
- 用户愿意为结果付费；
- 稳定性、UI、客服、数据安全、幻觉控制都能承担；
- 有持续获客和迭代能力。

判断标准：

- 用户现在是否已经为这个问题花钱？
- 这个工具是否比现有方案更便宜、更快或更可靠？
- 用户是否愿意持续使用，而不是试一次就走？
- 出错时责任边界是否清楚？
- UI 和操作是否足够低摩擦？

适合动作：

> 先做小范围付费验证，不要一上来做完整 SaaS。

## 飞书入口判断

如果一个 AI 应用涉及工作流、任务、审批、内容状态、多人协作，优先考虑飞书作为入口。

但要区分：

- 飞书 = 入口 / 中控台 / 状态板 / 人机协作界面；
- Obsidian = 长期知识资产库；
- Hermes / Agent = 调度与执行；
- 网站 / 公众号 / 小红书 = 对外发布与转化端。

不要把所有资产都塞进飞书，也不要把飞书当成最终知识库。

## 判断步骤

### 1. 先判断目标

问：

- 我是为了自己提效，还是为了卖给别人？
- 如果不赚钱，它是否仍然值得做？
- 如果要赚钱，谁会付钱？为什么现在就愿意付？

### 2. 再判断责任

问：

- 出 bug 谁承担？
- AI 幻觉是否会伤害用户？
- 是否需要人工验收？
- 是否需要日志、回滚、权限、备份？

### 3. 再判断体验

问：

- 用户第一次打开知道怎么用吗？
- 操作路径是否足够短？
- 结果是否可解释？
- 出错时是否有反馈？

### 4. 最后判断商业化

问：

- 是否有明确场景？
- 是否有付费意愿？
- 是否有复购/留存？
- 是否能低成本获客？
- 是否能持续维护？

## 输出

### 自用工具

动作：快速搭建，重点服务自己，不急着商业化。

### 交付组件

动作：先嵌入咨询/陪跑/服务里，作为提高交付效率的内部工具。

### 小范围产品

动作：找 3-5 个真实用户付费试用，验证价值和稳定性。

### 暂不产品化

动作：保留想法，先不要投入完整开发。

## 可调用提示

> 请用「AI应用开发双轨判断法」分析这个 AI 应用想法：
> 
> - 它更适合自用提效、交付组件、小范围产品，还是长期商业产品；
> - 它的 bug、幻觉、UI、维护责任在哪里；
> - 是否适合用飞书作为入口；
> - 最小验证动作是什么。

## 不适合做什么

- 不适合用“AI 降低门槛”来掩盖产品难度。
- 不适合把自用爽感直接等同于市场需求。
- 不适合一上来做大而全 SaaS。
- 不适合忽视用户视角和维护责任。

## 一句话

能做出来，只是开始；能被别人长期使用并付费，才是产品。
