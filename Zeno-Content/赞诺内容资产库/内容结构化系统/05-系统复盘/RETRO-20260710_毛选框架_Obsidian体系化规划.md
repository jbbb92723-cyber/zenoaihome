# 毛选框架分析：Obsidian 知识库体系化

> "没有调查就没有发言权。" —— 在给方案之前，先看清楚你现在到底有什么。

---

## 一、实事求是：现状诊断

### 当前真实状态

```
赞诺内容资产库/
├── ✅ AGENTS.md              [规则层：完整]
├── ✅ 内容结构化系统/
│   ├── ⚪ 00-系统配置/        [1个文件]
│   ├── ⚪ 04-产品服务/         [1个PRD]
│   ├── ⚪ 05-系统复盘/         [1个报告]
│   ├── ✅ 06-选题装配/         [~26个文件，最充实]
│   └── ⚪ 07-模板库/           [1个CTA模板]
│
│ ❌ 以下全部缺失：
│ ├── 01-原始素材区          [入口：空]
│ ├── 02-内容单元库          [加工层：空]  ← 这是核心问题
│ ├── 03-处理状态            [管理：无]
│ ├── 05-主题地图            [导航：无]
│ └── 07-脚本与工具          [自动化：无]
```

### 核心判断

> **你的知识库不是"需要体系化"——你的知识库是"出口跑通了，入口和加工层是空的"。**

类比：你建了一条生产线。产品能出厂（选题装配→网站文章），但原材料仓库（01原始素材）是空的，零件车间（02内容单元库）没有开工。你每次"生产"（写文章），都是从零开始徒手造——没有零件可以组装。

这不是结构问题。是**流转问题**：内容进去了（你写了26篇），但写完就放在那里，没有被拆回成可以复用的零件。

## 二、矛盾论：抓主要矛盾

### 表面的矛盾

- A："我的知识库不够体系化" ← 你以为的
- B："don哥的参考结构很完整" ← 你看到的
- C："我现在只有06选题装配是满的，其他都是空的" ← 真实的

### 主要矛盾

> **"你已经有能力产出高质量内容（26篇选题装配+5篇网站文章为证），但这些内容的'零件'——概念、案例、金句、判断框架——没有被拆成可复用的单元存回知识库。"**

主要矛盾不是"选哪个目录结构"——主要矛盾是：

```
产生内容 ──→ 发布 ──→ 结束（当前）
产生内容 ──→ 发布 ──→ 拆零件 ──→ 存入知识库 ──→ 下次写作时组装（想要的）
```

中间的"拆零件"和"存入"两步断了。

### 次要矛盾（先不管）

- 原始素材区怎么建？（没有持续的外部输入，建了也是空的）
- 主题地图怎么画？（内容单元不够多，地图画了也没东西可连）
- 脚本工具怎么写？（流程还没稳定，自动化太早）

→ **次要矛盾等主矛盾解决了自然会浮出水面。不要四面出击。**

## 三、持久战：三阶段规划

### 战略判断

**这是持久战，不是速决战。** 你一个人建知识库，参考结构有8个分区、几十个子目录。不要试图一周建完——那是教条主义。don哥的结构是"最终蓝图"，不是"本周TODO"。

### 三阶段

**第一阶段：战略防御——建"最小可行闭环"（现在→1个月）**

目标：让已经写出来的每篇内容，都能自动产生2-3个"零件"回到知识库。

```
写完一篇文章 
  → 发布到网站 ✓（已跑通）
  → 拆成零件 ✗（新增）
    ├── 概念单元[报价单=责任边界表] → 02-内容单元库/概念单元/
    ├── 案例单元[15万变23万的业主] → 02-内容单元库/案例单元/
    └── 方案单元[签约前自检清单]   → 02-内容单元库/方案单元/
  → 标记处理状态 ✗（新增）
```

这一阶段只做三件事：
1. 建 02-内容单元库 的5个子目录 + 模板
2. 把已发布的5篇文章拆成内容单元（每篇2-3个）
3. 养成习惯：每写一篇新文章，顺手拆2-3个零件

**第二阶段：战略相持——建"入口"和"连接层"（1-6个月）**

目标：有持续的外部素材输入 + 内容单元之间形成网络。

```
外部素材（业主提问、工地见闻、读书笔记）
  → 01-原始素材区（不带加工，扔进去就行）
    → 定期抽取 → 02-内容单元库
      → 主题地图连接 → 形成跨主题的判断网络
```

**第三阶段：战略反攻——半自动化生产（6-12个月后）**

目标：你问一个问题，AI从知识库检索相关概念+案例+方案，重组生成初稿，你只需要做最终判断。

```
用户提问："帮我看这份报价单"
  → AI检索：02-概念单元[责任边界表] + 02-案例单元[业主A vs B对比]
  → AI重组：生成初稿
  → 你审阅：判断、调整、发布
  → 新文章自动拆零件 → 回写知识库
```

## 四、战略上藐视，战术上重视

### 战略上藐视

don哥的参考结构有8个分区、几十个子目录。看起来吓人。

但你在过去两天已经证明了一件事：你能写出高质量的内容，能把它变成网站文章，能部署上线。你已经走通了最难的那一步——"从脑子里到纸上"。剩下的"把纸上的东西归类"——比那一步容易得多。

### 战术上重视

但分类这件事有陷阱。经典的分类陷阱：
- 分得太细→维护成本高→坚持不了→变成墓地
- 分得太粗→找不到东西→和没分一样

**don哥参考结构的核心价值不是目录名——是他定义了"5种内容单元"这个加工层。这不是为了分类而分类，是为了"未来检索和重组"而分类。**

## 五、群众路线：为谁建这个知识库？

### 你的"群众"是两个：

**1. 未来的你自己（人类检索者）**
- 需求："半年前我写过一篇关于付款节点的文章，里面有个案例，我想找出来用"
- 结构要求：能按主题找到 > 能按类型找到 > 能按时间找到

**2. Hermes AI助手（机器检索者）**
- 需求：执行 `search_files(pattern="责任边界", path="02-内容单元库")` 时能精准返回
- 结构要求：标准化命名 > 标准化frontmatter > 标准化存储

**不是为"看起来专业"建的，不是为"符合某种方法论"建的。是为这两个使用者建的。**

### 关键变化说明

| 变化 | 原因（毛选框架） |
|------|----------------|
| **02-内容单元库是第一阶段的根据地** | "集中优势兵力，各个歼灭"——选一个突破口。02是连接"入口"和"出口"的中枢，建好它，两头自然衔接。 |
| **05-主题地图、07-脚本工具暂缓** | "不要四面出击"——你只有一个人，同时建8个分区等于8个都没建好。先把一个分区做透了。 |
| **03-处理状态极简化** | don哥参考结构里03有9个子目录。当前你只需要2个文件：日志（做了什么）+ 清单（还没做什么）。等流转量大了再细分。 |
| **01-原始素材区用粗分类** | 4个子目录而不是更细。当前你还没有稳定的素材输入，太细的分类会让你找不到东西。粗一点，以后看了实际情况再调。 |
| **05-主题地图保留但永远不建** | 毛选说"先做起来再总结"。主题地图应该是在你积累了50+个内容单元之后自然浮现的——不是先画地图再填内容。 |

## 七、本周执行（只做三件事）

### 1. 建目录结构（15分钟）

```bash
mkdir -p "02-内容单元库/{CAS-案例单元,CON-概念单元,OPI-观点单元,QUE-问题单元,SOL-方案单元}"
mkdir -p "01-原始素材区/{业主提问,工地笔记,读书摘录,灵感碎片}"
mkdir -p "03-处理状态"
mkdir -p "04-模板"
echo "done"
```

### 2. 拆已发布的5篇文章（每篇15分钟）

把你过去两天写的5篇文章（108-113）拆成零件：

| 文章 | 可拆出的零件 |
|------|------------|
| 108-我是谁 | CON: 业主不敢问 vs 对方不说的裂缝 / CAS: 14.8万 vs 23万的业主对比 |
| 109-责任边界表 | CON: 报价单=责任边界表 / SOL: 6个模糊词自查清单 |
| 110-便宜报价最贵 | CAS: 省1.5万花8万的业主 / OPI: 判断报价看缺了什么不是看总价 |
| 111-按实际结算 | CON: 按实结算=金额栏空着的支票 / SOL: 5个缺口对照表 + 合同改写模板 |
| 112-付款节点 | OPI: 付款节奏=主动权安排 / SOL: 安全付款节奏表 |
| 113-信息差 | CON: 信息差=15-25%装修预算 / OPI: 知识易得，判断力稀缺 |

→ 拆出 **6个概念单元 + 4个案例单元 + 4个方案单元 + 3个观点单元** = 约17个零件

### 3. 写5个内容单元模板（30分钟）

参考don哥的模板，做成Obsidian模板文件放入04-模板/。每个模板包含：
- frontmatter（id/type/日期/标签/来源文章）
- 正文结构（一句话定义 + 展开 + 适用场景 + 相关单元链接）

## 八、一句话总结

> **你的知识库已经有一条跑通的出口（选题装配→网站文章）。现在要做的不是重建整个工厂——是在这条流水线上加一个"零件回收"环节：文章发出去了，顺手把里面的概念、案例、方案拆出来，扔回知识库。**

先把这个闭环跑一个月。一个月后看：02-内容单元库里有多少零件了？哪些零件被重复用到了新文章里？到那个时候，你自然知道下一步该建什么。

**先做了再总结。不要先总结再开始做。**
