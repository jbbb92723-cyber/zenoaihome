# 内容资产处理流程

## 核心原则

> **每发一篇新文章 → 顺手拆2-3个零件 → 扔回知识库。** 
> 不为完美，为养成习惯。不做完整分类，先做"能放回去的"。

## 标准流程

### 1. 写文章（06-选题装配 → 网站 articles.ts）
照常进行。已有流程不变。

### 2. 发布后拆零件（每篇15分钟）

对着刚发的文章，问自己三个问题：

| 问题 | 如果答案是YES | 对应单元类型 |
|------|-------------|------------|
| 这篇文章定义了什么核心概念？ | 抽出来 | → CON-概念单元 |
| 这篇文章讲了一个真实的故事/案例？ | 抽出来 | → CAS-案例单元 |
| 这篇文章给了一个可操作的清单/模板？ | 抽出来 | → SOL-方案单元 |
| 这篇文章表达了一个明确的判断/立场？ | 抽出来 | → OPI-观点单元 |
| 这篇文章回应了业主的什么真实问题？ | 记录下来 | → QUE-问题单元 |

### 3. 存零件（2分钟）

- 复制对应的模板
- 填一句话定义 + 来源链接
- 存入 02-内容单元库/对应子目录
- 在 03-处理状态/处理日志.md 加一行：`日期 | 文章xxx | 拆出：CON-xxx, CAS-xxx`

### 4. 不追求完美

**允许"半成品"存在。** 一个内容单元不需要填满模板的所有字段。最低要求：一句话定义 + 来源链接。其他字段以后用到时再补。

## 命名规则

- 概念单元：`CON-YYYYMMDD-编号_名称.md` （如 `CON-20260710-001_报价单=责任边界表.md`）
- 案例单元：`CAS-YYYYMMDD-编号_名称.md`
- 观点单元：`OPI-YYYYMMDD-编号_名称.md`
- 问题单元：`QUE-YYYYMMDD-编号_问题摘要.md`
- 方案单元：`SOL-YYYYMMDD-编号_名称.md`

## 铁律

1. **发布即拆**：文章上线后24小时内拆零件。超过24小时没拆，大概率永远不拆。
2. **不拆不发**：如果一篇文章没有任何可以拆成零件的东西——这篇文章可能太浅了。重新审视。
3. **宁缺毋滥**：不是所有内容都值得拆。如果一个观点、案例、方案在未来不太可能被复用→不拆。不要为了填目录而拆。
