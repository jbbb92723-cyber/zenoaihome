# Zeno 小红书每日内容包闭环 SOP

> 目标不是“一键洗稿爆款”，而是每天把小红书上的真实需求信号，转成 ZenoAIHome 可发布、可测试、可沉淀、可转化的内容包。

## 一句话定义

```text
小红书热点/对标/评论区痛点
→ 内容雷达筛选
→ Zeno 迁移判断
→ dbskill 标题与文案结构
→ baoyu-xhs-images 图片卡片
→ 每日平台内容包
→ 发布反馈回填
```

## 0. 系统边界

### 做什么

- 每天抓取小红书热点搜索、爆款标题、对标博主、评论区痛点。
- 只保留与 ZenoAIHome 主定位相关的信号：装修报价、签约前判断、预算超支、合同条款、材料型号、增项、付款节点、施工责任。
- 用 dbskill 做标题公式匹配、开头/内容结构诊断。
- 用 baoyu-xhs-images 生成小红书图片卡片。
- 每天输出一个「平台内容包」：标题、封面标题、图卡大纲、正文、标签、评论区引导、转化 CTA、素材来源、复盘字段。

### 不做什么

- 不照抄对标博主标题、图片、结构、案例。
- 不把“爆款监控”当成终点。
- 不制造装修焦虑，不写“不看就被坑惨”这类强恐吓语言。
- 不让 AI 默认完成最终人格化逐字稿；AI 先给可发布草稿和素材包，Zeno 可人工改最后一版。
- 不把小红书流量内容与服务资产割裂：每条内容都要能回到报价审核/签约前咨询/风险词库。

## 1. 每日输入源

### A. 热点搜索

目的：看今天平台上用户主动搜什么。

建议关键词池：

```text
装修报价、装修预算、装修避坑、装修合同、装修增项、半包报价、全包报价、装修公司报价、工长报价、硬装预算、材料清单、付款节点、按实结算、暂估价、漏项、增项单
```

采集字段：

| 字段 | 说明 |
|---|---|
| keyword | 搜索词 |
| rank | 热度/排序 |
| top_titles | 前 10-20 条标题 |
| repeated_words | 重复出现的词 |
| pain_signal | 暴露的痛点 |
| zeno_relevance | 0-5 |

### B. 爆款标题

目的：只借选题和心理触发，不借正文。

采集字段：

| 字段 | 说明 |
|---|---|
| original_title | 原标题 |
| platform_metrics | 点赞/收藏/评论/转发 |
| title_trigger | 数字/损失/身份/好奇/争议/结果承诺 |
| do_not_copy | 不能复制的具体表达 |
| reusable_mechanism | 可迁移机制 |

### C. 对标博主

目的：看稳定账号如何组织栏目和视觉，不抄单篇。

采集字段：

| 字段 | 说明 |
|---|---|
| blogger | 博主名 |
| niche | 装修避坑/报价/设计/施工/本地服务 |
| recent_posts | 近期内容 |
| strong_formats | 强势图文形式 |
| weak_points | Zeno 可以差异化的地方 |
| visual_traits | 图片风格特征 |

### D. 评论区痛点

目的：真正的选题来自评论区，不是标题。

优先保留这些评论：

```text
“报价看不懂”
“后面一直增项”
“合同没写清”
“这个算不算漏项”
“材料品牌没写型号”
“预算超了好多”
“付款后没人管”
“半包到底包含什么”
```

采集字段：

| 字段 | 说明 |
|---|---|
| user_words | 用户原话 |
| real_problem | 背后不会判断什么 |
| decision_scene | 签约前/比报价/施工中/验收 |
| service_question | 服务时该追问什么 |
| content_angle | 可转成什么小红书内容 |

## 2. 筛选评分：小红书高价值指数

每天只选 1-3 个可生产选题，避免被热点拖着跑。

```text
高价值指数 = 热度信号 + 评论痛点 + Zeno 相关度 + 服务转化 + 视觉可表达
```

| 维度 | 分值 | 判断问题 |
|---|---:|---|
| 热度信号 | 0-20 | 是否有明显点赞/收藏/搜索热度？ |
| 评论痛点 | 0-20 | 评论是否出现具体困惑和个人情况？ |
| Zeno 相关度 | 0-20 | 是否贴近报价风险/签约前判断？ |
| 服务转化 | 0-20 | 是否自然导向报价审核/咨询？ |
| 视觉可表达 | 0-20 | 是否能做成清单/对比/流程/风险图卡？ |

阈值：

- 80+：当天内容包候选。
- 60-79：进入选题池，等同类信号重复。
- 40-59：暂存观察。
- 40 以下：丢弃。

## 3. 每日处理流程

### Step 1：抓取与去重

输出：`daily_raw.md` 或飞书多维表记录。

- 抓热点搜索结果。
- 抓对标账号新内容。
- 抓高互动笔记标题和评论区高频痛点。
- 根据 URL / 笔记 ID / 标题相似度去重。
- 标记抓取异常：访问受限、IP 风险、验证码、空数据。

> 注意：当前环境里小红书网页经常出现 IP 风险限制，截图、手动导出的链接/文本、飞书/Coze 中转数据比直接浏览更稳。

### Step 2：内容雷达初筛

输出：`daily_radar_report.md`。

每天只推 3-5 条：

```markdown
# Zeno 小红书雷达战报｜YYYY-MM-DD

## 今日 3 条高价值信号
### 1）{信号标题}
- 来源：{搜索词/博主/评论区}
- 原始链接：{url}
- 为什么值得看：{一句话}
- 暴露痛点：{用户不会判断什么}
- 建议动作：做内容包 / 入痛点库 / 暂存

## 今日高潜选题
| 选题 | 用户场景 | 触发点 | 优先级 |
|---|---|---|---|

## 今日服务机会
- 报价审核：
- 签约前咨询：
- 风险词库新增：
```

### Step 3：选题拆解，不抄内容

对当天入选的 1 个选题，按这个顺序拆：

```text
表层选题
→ 评论区真实痛点
→ 隐藏误判
→ 底层机制
→ Zeno 迁移角度
→ 可转化服务问题
```

输出：`topic_deconstruction.md`。

### Step 4：调用 dbskill 做标题与文案骨架

#### 标题

调用 `dbs-xhs-title`，从 75 个公式中选 5-8 个，不自由乱起标题。

优先使用类型：

- 恐惧/损失型：避坑、警告、错误纠正。
- 数字锚定型：清单、步骤、检查表。
- 身份代入型：第一次装修、准备签合同的人。
- 好奇缺口型：装修公司/工长不会主动解释的报价细节。
- 互动/测试型：测一测你的报价单有没有风险。

标题验收：

- ≤ 20 字。
- 至少命中 2 个张力点：数字、悬念、具体场景、损失、身份。
- 不恐吓，不油腻，不像 AI。
- 能追溯到公式编号。

#### 文案

推荐结构：

```markdown
标题：
封面标题：
正文开头：一句场景化共鸣，不吓人
正文主体：3-5 个判断点，每点包含“怎么看 + 该问什么”
结尾：低压力 CTA
评论引导：邀请用户发一个看不懂的报价词/条款
```

Zeno 语气：

- 用“签字前多一个判断”，不用“你一定会被坑”。
- 用“这句话没写清，后面可能说不清”，不用抽象术语。
- 内容要导向判断，不只是情绪。

### Step 5：生成图片方案

调用 `baoyu-xhs-images` 前，先生成图片策略：

```yaml
image_strategy:
  count: 4-7
  style: bold / notion / sketch-notes / minimal
  layout: list / comparison / flow / dense
  palette: warm / macaron / default
  reference_mode: style-only / direct-ref-if-allowed
  innovation_point: 本次视觉区别于对标的地方
```

图片形式建议：

| 内容类型 | 推荐图片形式 | baoyu-xhs-images 参数 |
|---|---|---|
| 避坑清单 | 封面 + 风险列表 + 结尾清单 | `--preset warning` |
| 报价对比 | 左右对比：看似便宜 vs 实际风险 | `--preset versus` |
| 签约流程 | 3-5 步流程图 | `--preset hand-drawn-edu` |
| 知识解释 | 知识卡 / 概念拆解 | `--preset knowledge-card` |
| 自测互动 | 测试题 / 勾选表 | `notion + list` |

图片对标与创新规则：

- 对标只提取：色彩、信息密度、封面钩子、排版节奏。
- 创新必须改变至少 2 项：视角、结构、比喻、视觉符号、服务 CTA。
- 不复制对方封面构图、标题句式、案例和图标组合。
- 每张图的 prompt 必须先保存到 `prompts/`。
- 第一张图生成后作为 anchor，后续图用第一张作为参考，保持风格一致。

### Step 6：形成每日平台内容包

输出目录建议：

```text

├── 00_来源与雷达记录.md
├── 01_选题拆解.md
├── 02_标题池.md
├── 03_小红书文案.md
├── 04_图片方案.md
├── 05_发布包.md
└── images/
```

`05_发布包.md` 模板：

```markdown
# 小红书发布包｜{日期}｜{主题}

## 1. 推荐发布标题

{标题}

- 公式来源：dbs-xhs-title #{编号}
- 选择理由：

## 2. 封面标题

{封面标题}

## 3. 正文

{正文}

## 4. 图片顺序

1. {封面图} - {作用}
2. {内容图} - {作用}
3. {内容图} - {作用}
4. {结尾图} - {作用}

## 5. 标签

#装修避坑 #装修报价 #装修预算 #装修合同 #装修增项 #半包装修 #全包装修 #装修小白 #签约前必看 #赞诺Zeno

## 6. 评论区引导

你家报价单里有没有“按实际发生结算 / 暂估 / 另计 / 不含”？可以评论区发一个词，我帮你判断该追问什么。

## 7. 低压力转化 CTA

如果你已经拿到报价单，但不确定哪些地方该问清，可以先做一轮签约前报价风险初筛。

## 8. 发布检查

- [ ] 标题 ≤ 20 字
- [ ] 封面标题可读
- [ ] 正文没有恐吓式表达
- [ ] 有评论引导
- [ ] 有服务入口但不过度销售
- [ ] 已记录来源，不抄原文
```

### Step 7：发送、移动端发布与归档

每天发送到微信/飞书：

```markdown
# 今日小红书内容包｜YYYY-MM-DD

## 今日推荐发布
- 标题：
- 选题来源：
- 用户痛点：
- 发布优先级：A/B/C

## 内容包
- 正文：见附件/链接
- 图片：{N} 张
- 标签：
- 评论引导：

## 需要 Zeno 人工确认
1. 标题是否保留？
2. CTA 是否太硬？
3. 图片风格是否符合账号？
```

#### 移动端发布解决方案

痛点：内容和图片生成在本地电脑，但 Zeno 不一定在电脑旁，无法顺手发布小红书。

推荐采用“三层交付”：

```text
本地生成
→ 微信/飞书推送发布包
→ 手机保存图片 + 复制文案
→ 小红书 App 手动发布
```

##### 方案 A：微信直达发布包（最轻量，推荐 v0）

每天生成内容包后，Hermes 直接把以下内容发到微信：

- 6 张图片作为原图/附件。
- 标题 Top 3。
- 最终正文。
- 标签。
- 评论区引导。
- CTA。
- 发布检查清单。

手机端操作：

1. 长按保存图片到相册。
2. 复制标题、正文、标签。
3. 打开小红书 App。
4. 选择图片并发布。

优点：简单、稳定、符合小红书移动端发布习惯。  
缺点：仍需人工点发布。

##### 方案 B：飞书内容审批台（适合 v1）

每天把内容包写入飞书多维表/文档：

| 字段 | 说明 |
|---|---|
| 发布状态 | 待审 / 待发 / 已发 / 复盘 |
| 图片附件 | 生成图卡 |
| 标题 | 推荐标题 |
| 正文 | 可复制正文 |
| 标签 | 可复制标签 |
| CTA | 服务入口 |
| 审核意见 | Zeno 手机端修改 |

手机端操作：

1. 在飞书看当天内容包。
2. 改标题或 CTA。
3. 标记“待发”。
4. 保存图片到相册，再去小红书发。

优点：适合审批、多人协作、历史管理。  
缺点：多一步飞书操作。

##### 方案 C：云盘同步相册包（适合图片很多时）

Hermes 将图片同步到一个云盘目录，例如：

```text
小红书待发布/YYYY-MM-DD-{topic}/
```

手机端通过云盘 App 批量保存图片，再复制文案发布。

优点：图片管理更清楚。  
缺点：仍需配置云盘同步链路。

##### 暂不推荐：完全自动代发小红书

原因：

- 小红书没有稳定公开发布 API。
- 自动化网页登录/模拟发布容易遇到验证码、风控、账号风险。
- 对 Zeno 来说，发布前的人审很重要：标题、CTA、语气、图片文字都需要最后看一眼。

所以 v0/v1 的正确目标不是“全自动发布”，而是：

```text
让 Zeno 在手机上 2 分钟完成发布
```

归档位置：

## 4. 每周复盘

每周看 3 件事：

1. 哪类标题获得点击/收藏？
2. 哪类评论暴露服务机会？
3. 哪种图片形式更适合 ZenoAIHome：warning、notion、sketch-notes、comparison？

复盘字段：

| 字段 | 说明 |
|---|---|
| publish_url | 发布链接 |
| views | 浏览 |
| likes | 点赞 |
| saves | 收藏 |
| comments | 评论 |
| follows | 关注 |
| inquiries | 私信/咨询 |
| best_comment | 最有价值评论 |
| next_asset | 要沉淀的服务问题/方法卡 |

## 5. 最小可用版本

### v0：手动半自动，先跑 7 天

- Zeno 每天给 3-5 条小红书链接/截图/评论区文本。
- Hermes 输出 1 个完整内容包。
- baoyu-xhs-images 生成 4-6 张图。
- 内容包发微信/飞书，不自动发布。

### v1：半自动雷达

- 固定 10-30 个对标来源。
- 每天定时生成雷达战报。
- Zeno 选择 1 个选题。
- Hermes 生成内容包 + 图片。

### v2：自动候选内容包

- 每天自动生成 1 个候选内容包。
- Zeno 只做发布前审核。
- 发布数据每周回填。

### v3：内容中台闭环

- 来源库、内容库、痛点库、选题库、拆解库打通飞书/Obsidian。
- 高价值痛点自动进入服务 SOP 和网站 FAQ 候选。
- 小红书内容不再只是发帖，而是服务转化入口。

## 6. 关键验收标准

7 天试运行成功标准：

- 每天至少产出 1 个可发布内容包。
- 每个内容包都能追溯到来源信号和 dbskill 公式。
- 每组图片都有独立 prompt 和可复用风格记录。
- 至少沉淀 5 条真实评论痛点。
- 至少出现 1 个可导向报价审核/签约前咨询的服务问题。

## 一句话提醒

**每天不是追一个爆款，而是把一个真实痛点变成：一条小红书内容、一个服务追问、一个判断资产。**
