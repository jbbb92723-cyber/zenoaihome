# 登录数据库与主题系统实施说明

> 本文档记录本次系统改造的技术决策和实现细节。

---

## 1. 为什么选 Supabase Postgres

- 免费额度 500MB，足够个人站点
- 同时提供 Pooled（6543 端口，用于 Serverless）和 Direct（5432 端口，用于 Migrate）连接
- 无需自建数据库服务器

## 2. 为什么用 Prisma 而不是 Drizzle

- Prisma 生态更成熟，Auth.js 官方 PrismaAdapter 稳定
- 自动生成类型安全的客户端
- `prisma migrate` 管理数据库迁移更直观

## 3. 为什么不用 Supabase Auth

- Supabase Auth 会绑定到 Supabase 平台，迁移成本高
- Auth.js (next-auth v5) 是 Next.js 生态标准方案
- 自行管理密码哈希和验证码逻辑，完全可控

## 4. 邮箱注册登录流程

1. 用户在注册页输入邮箱 → 点击「发送验证码」
2. 后端生成 6 位数字验证码，bcrypt 哈希后存入 `VerificationCode` 表
3. 通过 Resend API 发送验证邮件
4. 用户输入验证码 + 密码 → 后端校验验证码（最多 5 次尝试，10 分钟过期）
5. 校验通过后创建用户（密码 bcrypt 哈希存储）

## 5. 为什么用验证码而不是 Magic Link

- 用户习惯更好（不需要切换到邮箱点击链接）
- 验证码有明确的有效期和尝试次数限制
- 适用于注册、忘记密码、修改密码三个场景

## 6. Google 登录实现

- 使用 Auth.js 内置 Google Provider
- 开启 `allowDangerousEmailAccountLinking`，允许同邮箱的 Google 和密码账号关联
- 回调地址：`https://zenoaihome.com/api/auth/callback/google`

## 7. 密码安全策略

- 最少 8 位，必须同时包含字母和数字（zod 校验）
- bcrypt 哈希存储，默认 10 轮
- 修改密码需要邮箱验证码确认

## 8. 管理员系统为什么独立

- 管理员只有站长一人，无需复杂角色系统
- 通过 `ADMIN_PASSWORD` 环境变量设置密码
- Cookie 使用 HMAC-SHA256 签名，7 天过期
- 完全独立于普通用户的 Auth.js 登录系统

## 9. JWT 会话策略

- 使用 JWT 而不是数据库 Session，减少数据库查询
- JWT 中存储 `id`、`role`、`provider` 字段
- 30 天有效期

## 10. 验证码安全措施

- 每个邮箱 60 秒冷却时间
- 验证码 10 分钟过期
- 最多 5 次错误尝试
- 验证码使用 bcrypt 哈希存储（不存明文）
- 使用后标记 `consumedAt`，不可重复使用

## 11. 暗色主题实现

- 使用 `next-themes` 库，`class` 策略
- CSS 变量在 `:root`（亮色）和 `.dark`（暗色）中定义
- Tailwind 语义颜色（canvas, surface, ink, stone, border）自动响应主题切换
- 暗色调为温暖暗灰（#1C1A17），非纯黑，保持品牌调性

## 12. 删除的旧模块

- Neon Serverless Postgres → 替换为 Supabase Postgres
- Drizzle ORM → 替换为 Prisma
- IDC Flare OAuth → 删除
- 微信扫码登录 → 删除
- Magic Link → 替换为验证码
- ADMIN_EMAILS / ADMIN_USER_IDS → 替换为 ADMIN_PASSWORD

## 13. 保留的模块

- md2wechat 排版工具（API 转换 + 草稿推送）
- 微信公众号草稿箱 API（WECHAT_APPID / WECHAT_APP_SECRET）
- 火山引擎豆包图片生成（VOLCENGINE_*）

## 14. 数据库表结构

| 表 | 用途 |
|---|---|
| User | 用户主表，含 passwordHash |
| Account | OAuth 账号关联（Google） |
| Session | Auth.js 内部（JWT 模式下基本不用） |
| VerificationToken | Auth.js 内部 |
| VerificationCode | 邮箱验证码，含 codeHash、尝试次数、过期时间 |
