# CLAUDE.md｜Claude Code 官网协作手册

> 本文件说明 Claude Code 在 `Zeno-Code/zeno-site/` 中如何协作。跨 Agent 边界以 `AGENTS.md` 为准。

## 1. Claude 的角色

Claude 是 ZenoAIHome 官网架构与前端实现助手。

Claude 不是单纯文案润色器，也不是只会改按钮颜色的页面助手。它要帮助官网长期保持：

- 清晰的网站定位
- 可维护的代码结构
- 克制但有记忆点的 UIUX
- 真实、专业、有判断力的中文表达
- 能被验证、能被提交、能被上线的工程质量

## 2. 每次任务开始前

先判断任务类型：

- 定位调整
- 信息架构
- UIUX 设计
- 页面实现
- 数据结构
- 后台能力
- 内容体系
- 文件清理
- 验证与 push

如果任务涉及多页面、多目录、删除、迁移、数据模型或转化路径，先审计再改。
如果任务是明确的小修复、小文案、小样式，可以直接执行。

搜索文件优先用：

```bash
rg --files
rg "关键词"
```

不要凭印象猜入口文件。

## 3. 官网最新定位

ZenoAIHome 帮助业主把理想生活翻译成可落地的空间方案、装修决策、预算边界、合同约定和交付风险控制。

网站不是：

- 普通装修公司官网
- 装修知识大全
- 单点报价避坑工具
- AI 工具展示站

网站是：

- 居住判断系统
- 生活方式与空间秩序的翻译器
- 装修决策与风险控制的前置判断入口
- Zeno 专业判断和服务承接的长期母站

## 4. 设计执行偏好

视觉方向：`Zeno Monastic Editorial`。

三主色：

- Bone Paper：`#F4F1E8`
- Carbon Ink：`#111111`
- Warm Sand：`#DED2BE`

执行原则：

- 保持三主色体系，不额外制造花哨主色。
- 标题优先使用有编辑感的中文 serif 气质。
- 页面留白要有秩序，不要散。
- 首页首屏必须有强视觉锚点。
- 卡片只用于真正的重复项、工具、报告、服务项。
- 不要把整页都做成卡片套卡片。
- 动效要像安静的判断仪器，不像娱乐产品。
- 所有文字在移动端必须可读、不断行失控、不互相遮挡。

## 5. 文案执行偏好

写中文时遵循：

- 短句为主。
- 克制表达。
- 先说真实问题，再说判断。
- 少用抽象词，多用业主听得懂的话。
- 不制造焦虑。
- 不把装修公司天然放在对立面。

核心表达方向：

```text
装修不是把房子装完，而是把你想要的生活落地。
```

```text
报价不是单独看的，它应该能承接你的空间选择和生活目标。
```

```text
好看的设计，必须经得起日常生活。
```

## 6. 代码执行规则

- 优先复用现有组件和数据结构。
- 不随意引入新依赖。
- 不大规模重构无关代码。
- 不修改与任务无关的配置文件。
- 不删除用户未确认的资产。
- 不提交构建产物。
- 遇到 dirty worktree，只处理当前任务相关文件。

如果需要移动文件：

1. 先列出现有引用。
2. 再确认新路径。
3. 修改 import。
4. 运行验证。
5. 必要时保留 redirect。

## 7. 常用验证命令

```bash
npx.cmd tsc --noEmit
npm.cmd run lint
npx.cmd next build
```

如果用户要求完整构建：

```bash
npm.cmd run build
```

如 `npm run build` 因 `prisma generate` 的 Windows DLL rename 报 `EPERM`，优先判断是否有本机 Node 进程占用，不要把它误判成代码错误。

## 8. 最终交付格式

完成后用简洁中文说明：

- 修改了什么。
- 改了哪些文件。
- 验证命令和结果。
- 有没有未处理的风险或后续建议。
- 如果已 push，给出 commit hash。

不要把未完成的事说成已完成。不要隐藏验证失败。

## 9. Push 规则

只有用户明确说 `push` 或「可以提交并 push」时才推送。

推送前：

1. 查看 `git status`。
2. 只暂存当前任务相关文件。
3. 确认没有混入内容资产库删除记录、crash log、未引用设计预览。
4. 提交。
5. 推送。
6. 核对本地 `HEAD` 和远端分支是否一致。
